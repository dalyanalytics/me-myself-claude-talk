Mistake #1: Package Confusion

Claude doesn‚Äôt read the docs‚Äîjust like you when you‚Äôre rushing

  • What happened: Asked Claude about {shinyfa} ‚Üí it hallucinated ‚ÄúFont Awesome icons‚Äù
  • The real error: Like installing a package and guessing without reading vignettes/examples
  • It‚Äôs not just new packages: Even with {ggplot2}, Claude confuses function arguments
  • Why this happens: You‚Äôre experienced, busy, assume things should come easy, so you circumvent
  • The lesson: This forces you to actually read documentation and verify behavior
  • Fix strategy: Feed Claude the documentation URL, then it self-corrects

This mistake makes you better at evaluating package quality and knowing where good docs live.

I‚Äôm going to share some mistakes that happened even building this talk with Claude Code.

Claude doesn‚Äôt always read the docs‚Äîjust like humans rushing through things. When I had Claude structure the initial slides, it completely hallucinated what the shinyfa package does. It assumed ‚ÄúFA‚Äù meant Font Awesome icons. It didn‚Äôt even install the package‚Äîjust guessed about vignettes and examples.

This happens even with established packages like ggplot2. Experience breeds assumptions, and you skip verification.

The lesson: feed Claude documentation URLs so it can self-correct. This actually makes you better at evaluating where good docs live.

Mistake #2: Data Structure Assumptions

Claude assumes instead of checking‚Äîsound familiar?

  • What Claude does: Assumes column names, data types, relationships
  • Example: Used avilist_global (doesn‚Äôt exist) instead of avilist_2025
  • Example: Assumed lowercase order_name when actual column is Order
  • Example: Infers connections between columns instead of asking for lookup tables
  • Why this happens: Experience breeds assumptions; you skip verification when busy
  • The lesson: Use mcptools to give Claude actual R session context
  • Fix strategy: names(data), str(data), or let mcptools read your environment

This mistake teaches you proper data inspection habits‚Äîskills you use constantly.

Claude loves to infer from column names and data types. But sometimes it‚Äôs just wrong.

Building the AviListR slide, Claude called the dataset avilist_global‚Äîwhich doesn‚Äôt exist. It‚Äôs avilist_2025. It also assumed snake_case column names when they actually match the original Excel file‚Äôs title case.

Experience breeds assumptions. You skip verification. ‚ÄúWell, snake_case is commonplace, right?‚Äù

The lesson: leverage MCP tools to give Claude actual access to your R session. If packages are loaded, if objects exist, let MCP integrate and read your environment.

This mistake teaches proper data inspection habits‚Äîskills we should all be using constantly as data practitioners.

Mistake #3: Over-Abstraction

Why is Claude (or you) always doing it the hard way?

  • Claude‚Äôs tendency: Always reaches for complex solutions first
  • Example: Tried 3 different Sankey packages with hallucinated functions
  • Example: Creates nested functions when a simple dplyr chain works
  • Why this happens: You want to show expertise, build ‚Äúproper‚Äù systems, over-engineer
  • The lesson: Simple is better. Ask ‚ÄúWhy are you doing it the hard way?‚Äù
  • Fix strategy: Ask for multiple options, choose the simplest that works

This mistake teaches you that simplicity is a skill‚Äîrecognizing unnecessary complexity.

This is my biggest gripe with Claude. It always reaches for the complex solution first. Always trying to be overly clever.

Building this slide deck, Claude tried creating an intricate Sankey diagram and just started making up functions. Nested functions everywhere when a simple dplyr chain would work.

I think it‚Äôs pulling from examples where people are showing expertise, building proper systems‚Äîwhich sometimes leads to over-engineering. Missing the focus: simple is often better.

So I ask Claude a very direct question: ‚ÄúWhy are you doing it this way?‚Äù Often, because I‚Äôve spent so much time without AI, I do know the solution. I know the exact packages I want to use.

I ask why, feed it different options, and select the simplest way‚Äîthe way I would actually do it. Recognizing unnecessary complexity has been key to guiding this partnership.
